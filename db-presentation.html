01

PostgreSQL Internals

How Postgres stores, retrieves, and protects your data.

Architecture at a Glance

Process Model

Process-per-connection. Postmaster forks a backend for each client. 5-10 MB per connection.

Shared Memory

Shared buffers (page cache), WAL buffers, lock tables. Set shared_buffers to 25% of RAM.

Background Workers

WAL writer, checkpointer, autovacuum, background writer, stats collector.

MVCC: Multi-Version Concurrency Control

  • UPDATE never overwrites — creates a new tuple version, marks old one dead
  • Each transaction sees a snapshot — reads never block writes, writes never block reads
  • Dead tuples accumulate until VACUUM reclaims the space
  • Table bloat if autovacuum can't keep up with update rate
UPDATE row
→
Old tuple: xmax = current_txid
→
New tuple: xmin = current_txid
→
VACUUM cleans old tuple later

Write-Ahead Log (WAL)

1. WAL Buffer
Write log record to memory
2. fsync WAL
Flush to pg_wal/ on disk
3. COMMIT returns
Client gets OK
4. Checkpoint
Dirty pages → data files

WAL guarantees durability and enables replication. Same mechanism for both.

VACUUM: The Garbage Collector

TypeWhat It DoesLocks?
VACUUMMarks dead tuple space reusable. Does NOT return space to OS.None (concurrent)
VACUUM FULLRewrites entire table. Returns space to OS.EXCLUSIVE (blocks all)
VACUUM ANALYZEVACUUM + update planner statisticsNone

If autovacuum falls behind → table bloat → slow queries → TX ID wraparound (Postgres shuts down!)

Transactions & Isolation

Making concurrent access safe.

Isolation Levels

LevelDirty ReadNon-RepeatablePhantomWrite Skew
READ COMMITTED defaultNoYesYesYes
REPEATABLE READNoNoNo*Yes
SERIALIZABLENoNoNoNo

*Postgres REPEATABLE READ uses snapshot isolation — prevents phantoms too (stronger than SQL standard).

Locking Strategies

Pessimistic

SELECT ... FOR UPDATE

  • Lock the row before reading
  • Other transactions wait
  • Best for: high contention, short txns
  • Variant: SKIP LOCKED for job queues
Optimistic

WHERE version = N

  • Read without locking
  • On write, check version unchanged
  • Retry on conflict (rowcount = 0)
  • Best for: low contention, longer txns

Making queries fast as data grows.

Index Superpowers

Partial Index

Only index rows matching a condition. Tiny and fast.

CREATE INDEX idx_pending
ON orders(created_at)
WHERE status = 'pending';
Covering Index

Include columns to avoid heap lookups entirely.

CREATE INDEX idx_cover
ON orders(user_id, status)
INCLUDE (total, created_at);

04

Advanced SQL

Window functions, CTEs, JSONB, full-text search.

Window Functions

Compute aggregates without collapsing rows. Run AFTER WHERE/GROUP BY, BEFORE ORDER BY.

Ranking

ROW_NUMBER(), RANK(), DENSE_RANK()

Running Totals

SUM() OVER (ORDER BY ...)

Neighbors

LAG(), LEAD() for prev/next values

-- Top 3 orders per user
WITH ranked AS (
  SELECT *, ROW_NUMBER() OVER (
    PARTITION BY user_id ORDER BY total DESC
  ) AS rn FROM orders
)
SELECT * FROM ranked WHERE rn <= 3;

JSONB & Full-Text Search

JSONB
  • @> containment (GIN indexed)
  • ->> extract as text
  • jsonb_set() update nested values
  • Use for flexible metadata, not main columns
Full-Text Search
  • tsvector + tsquery
  • setweight() to boost title over body
  • ts_rank() for relevance sorting
  • GIN index for sub-ms lookups

SQLAlchemy & Alembic

The Python database layer.

SQLAlchemy 2.0 Key Concepts

Engine + Session
  • create_async_engine — connection pool
  • async_sessionmaker — unit of work
  • expire_on_commit=False — critical for async
  • Separate engines for read/write
Models
  • mapped_column() with type hints
  • lazy="raise" — prevent N+1
  • selectinload for collections
  • joinedload for single objects

Zero-Downtime Migrations

  • Add column as NULL first, backfill, then add NOT NULL
  • Use CREATE INDEX CONCURRENTLY (no write lock)
  • Never rename columns — add new, backfill, drop old
  • Never ALTER COLUMN TYPE on large tables (rewrites table)
  • Autogenerate misses partial indexes and CHECK constraints

Async Replication

Primary commits without waiting. Fast writes. Replica may lag (ms to sec). Risk: data loss on primary crash.

Sync Replication

Primary waits for replica ACK before commit. No data loss. Writes slower (network RTT).

07

MongoDB

Flexible documents, aggregation, Atlas Search.

When MongoDB Wins

Use MongoDB
  • Schema varies per document
  • Deeply nested, read as a unit
  • High write throughput, eventual consistency OK
  • Native horizontal sharding
Stay with PostgreSQL
  • Well-defined, stable schema
  • Complex JOINs across tables
  • Strict ACID transactions
  • Financial/inventory data

Schema Design: Embed vs Reference

StrategyWhenExample
EmbedRead together, bounded, one-to-fewOrder + line items, user + addresses
ReferenceQueried independently, unboundedUser → orders, product → reviews

16 MB document limit! Embedding unbounded arrays (all reviews) will hit this.

Aggregation Pipeline

$match
→
$unwind
→
$lookup
→
$group
→
$sort
→
$project
  • $match first — uses indexes, reduces data early
  • $unwind flattens arrays — one doc per element
  • $lookup is a LEFT JOIN to another collection
  • $group is GROUP BY with $sum, $avg, $addToSet

Redis

In-memory speed, rich data structures, production patterns.

Production Patterns

Cache-Aside

Check Redis → miss? → query DB → store in Redis. Invalidate on write. Prevent stampede with locks.

Rate Limiting

Sliding window with sorted sets + Lua script. Atomic operation, no race conditions.

Distributed Lock

SET NX EX + Lua release. Token ownership prevents releasing another's lock.

Pub/Sub vs Streams

Pub/Sub: fire-and-forget. Streams: durable log with consumer groups (like lightweight Kafka).

Persistence & HA

MechanismHowTrade-off
RDB (snapshots)Periodic fork + dumpFast recovery, may lose last 5 min
AOF (append log)Log every writeMinimal loss, slower recovery
RDB + AOFBothBest of both worlds (recommended)

Redis is a cache, NOT a primary database. Always be able to rebuild from your source of truth.

CAP Theorem

Network partitions are inevitable. Choose: Consistency or Availability.

CP (Consistent)

PostgreSQL, MongoDB, etcd

Returns error rather than stale data. Writes may block.

AP (Available)

Cassandra, DynamoDB, CouchDB

Returns potentially stale data. Accepts all writes. Reconciles later.

Key Takeaways

PostgreSQL
  • MVCC, WAL, VACUUM — understand the internals
  • Indexes: B-tree, GIN, BRIN, partial, covering
  • EXPLAIN ANALYZE is your best debugging tool
  • Connection pooling (PgBouncer) + read replicas
MongoDB
  • Embed for bounded, read-together data
  • Reference for unbounded, independent data
  • Aggregation pipeline: $match first, always
  • Atlas Search for Lucene-powered search
Redis
  • Use the right data structure, not just strings
  • Lua scripts for atomic multi-step operations
  • Cache-aside + invalidation + stampede protection
  • Redis is a cache, not a primary database
System Design
  • Start simple: PG + Redis handles most apps
  • Scale reads first (replicas, caching)
  • Shard as a last resort
  • Each database for its strength: polyglot persistence
