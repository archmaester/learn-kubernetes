01

The Problem Docker Solves

Dependency hell — and the three eras of deployment

Start with the story everyone has lived through at least once as an ML engineer.

A Story Every ML Engineer Knows

You train a sentiment classifier on your MacBook. 94% accuracy, sub-50ms inference. You hand it to the platform team. An hour later: "It doesn't work."

The Chain of Failures
  • Their server has Python 3.9 — your code uses match (3.10+)
  • Install 3.11 — NumPy can't find libopenblas
  • Fix that — PyTorch complains about CUDA 11.8 vs 12.1
  • Three days later: polluted production server with pinned system libraries

This is dependency hell. Docker was built to eliminate it.

ML dependency problems are worse than typical web dev because you also have CUDA versions, cuDNN, C extensions, hardware-specific optimizations.

Three Eras of Deployment

EraMechanismOverheadBoot timePain point
Bare Metal One app per server 10–15% utilization Weeks to provision Snowflake servers, configuration drift
Virtual Machines Hypervisor carves hardware 1–2 GB per VM 30–60 seconds Duplicate OS kernels, heavy images (10–20 GB)
Containers Now Shared kernel, isolated env Megabytes Milliseconds Weaker isolation than VMs (shared kernel)
The key insight: applications don't need a separate kernel — they just need an isolated view of filesystem, processes, and network.

The Apartment Building Analogy

Virtual Machines = Houses
  • Each house has its own foundation, plumbing, electrical, heating
  • Complete isolation
  • Extremely expensive to replicate
Containers = Apartments
  • Share the building's foundation (kernel) and infrastructure
  • Each unit has private space (filesystem, processes, network)
  • Efficient — shared infrastructure, independent living

Trade-off: if the building's foundation has a crack (kernel vulnerability), it affects every apartment.

Docker didn't invent containers. Linux has had namespaces and cgroups since 2008. Docker made them accessible and standardized the format (OCI).

Key fact

The CLI communicates via Unix socket (/var/run/docker.sock). The daemon and CLI don't need to be on the same machine.

Why Kubernetes "dropped Docker" in 2021

K8s uses containerd directly, skipping Docker's daemon layer. Docker was never required — it was just one path to containerd.

runc starts the container then exits. containerd-shim keeps it alive after runc exits — so you can upgrade containerd without killing running containers.

Three Linux Kernel Primitives

Namespaces

What a container sees

  • pid — process IDs
  • net — network stack
  • mnt — filesystem
  • uts — hostname
  • ipc — shared memory
  • user — UID/GID
cgroups v2

How much it uses

  • Memory caps → OOM killer
  • CPU quota
  • I/O bandwidth
  • Exit code 137 = killed by OOM
OverlayFS

Copy-on-write filesystem

  • Read-only image layers stacked
  • Writable container layer on top
  • Modified files copied up
  • Deleted on container removal
On macOS, there's a hidden Linux VM (Apple Virtualization framework) running underneath everything — hence the extra I/O latency for file sharing in dev.
03

Images and Layers

The packaging model — content-addressable, shared, versioned

What IS a Docker Image?

A stack of read-only filesystem layers, each identified by a SHA256 digest. Together they form a complete filesystem snapshot.

Creates a layer
  • RUN
  • COPY
  • ADD
Metadata only (no layer)
  • ENV
  • EXPOSE
  • CMD / ENTRYPOINT
  • WORKDIR

100 containers from the same image share the same read-only layers. Zero wasted disk space.

Layer Cache — Order Matters

If a layer changes, all subsequent layers are invalidated.

Bad — cache-busting order
COPY . .          # changes every commit
RUN pip install   # always rebuilds
Good — cache-friendly order
COPY requirements.txt .   # rarely changes
RUN pip install           # cached unless deps change
COPY . .                  # source code last
Think of it like a stack of pancakes. Remove one, everything above it must be re-stacked.

Base Image Choice — Size Is a Real Cost

ImageSizeCVEs (approx)Best for
python:3.12 ~1 GB 100+ Development, debugging
python:3.12-slim Recommended ~150 MB ~20 ML workloads, FastAPI, most Python apps
python:3.12-alpine ~50 MB ~5 Simple CLIs — not ML (musl libc breaks NumPy/SciPy)
gcr.io/distroless/python3 ~30 MB Minimal Production-only, no shell (max security)

A 2 GB image takes 30–60s to pull. A 150 MB image takes ~5s. At scale, that's minutes of deploy lag.

04

Writing Dockerfiles

From bad to production-grade — 7 failure modes and how to fix them

The Bad Dockerfile

# Every problem in 6 lines
FROM python:3.12                          # 1GB, 100+ CVEs
WORKDIR /app
COPY . .                                  # code first → cache miss every commit
RUN pip install -r requirements.txt       # rebuilds on every code change
RUN apt-get update && apt-get install -y curl vim htop   # dev tools in prod
CMD python main.py                        # shell form → no SIGTERM handling
Problem 1

Wrong base image — massive size & CVE surface

Problem 2

COPY order busts cache on every code change

Problem 3

Shell form CMD breaks graceful shutdown

The Production Dockerfile

FROM python:3.12-slim-bookworm AS builder
RUN apt-get update && apt-get install -y --no-install-recommends gcc && \
    python -m venv /opt/venv
COPY requirements.txt .
RUN /opt/venv/bin/pip install --no-cache-dir -r requirements.txt

FROM python:3.12-slim-bookworm AS runtime
COPY --from=builder /opt/venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
WORKDIR /app
RUN groupadd --gid 1000 appuser && useradd --uid 1000 --gid appuser appuser
COPY --chown=appuser:appuser . .
USER appuser
HEALTHCHECK --interval=30s --timeout=5s \
    CMD ["curl", "-f", "http://localhost:8000/health"]
CMD ["python", "-m", "uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
Multi-stage build: builder has gcc and build tools. Runtime stage only has the venv. Stripping build tools can save 200-400MB.

The Production Dockerfile Checklist

  • Slim or distroless base image (not full Debian)
  • Multi-stage build — build tools excluded from final image
  • pip install --no-cache-dir — saves 50–200 MB
  • Dependencies copied and installed before application code
  • Non-root user created and activated before CMD
  • Exec form CMD ["python", ...] — PID 1 receives SIGTERM directly
  • HEALTHCHECK defined
  • Secrets via BuildKit mounts, never ENV or ARG
  • .dockerignore excludes .git, __pycache__, .env, test data
ENV and ARG values are visible in docker inspect and docker history — anyone with image access can read them. Use --mount=type=secret for build-time secrets.

Lifecycle, networking, volumes, and resource limits

Stopping
  • docker stop → SIGTERM → waits 10s → SIGKILL
  • docker kill → SIGKILL immediately
  • Exit code 143 = graceful shutdown (SIGTERM)
  • Exit code 137 = OOM kill (SIGKILL)
Restart Policies
  • no — never restart (default)
  • on-failure[:n] — on non-zero exit
  • always — even after manual stop
  • unless-stopped — not after explicit stop ← use this

Networking — Use Custom Bridge Networks

Default bridge network

No DNS — containers can only reach each other by IP. IPs change on restart. Scripts break.

Custom bridge network

Automatic DNS by container name. redis://redis:6379 just works.

docker network create ml-services
docker run -d --name redis --network ml-services redis:7-alpine
docker run -d --name api  --network ml-services -p 8080:8000 my-api:v1
# Inside api container: redis is reachable as hostname "redis"

Storage — Three Types

TypeManaged byPersists?Best for
Named volume Docker Yes Databases, model weights, production data
Bind mount Host filesystem Yes Dev hot-reload, sharing host files
tmpfs Host RAM No Temp files, in-memory secrets, /tmp 
docker run -d -v model-cache:/app/cache my-api:v1         # named volume
docker run -d -v $(pwd)/src:/app/src my-api:v1             # bind mount (dev)
docker run -d --tmpfs /tmp:size=100m my-api:v1             # tmpfs
06

Docker Compose

Multi-container stacks as declarative YAML

Anatomy of a Production Compose File

services:
  api:
    build: { context: ., target: runtime }
    ports: ["8080:8000"]
    environment:
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      redis:
        condition: service_healthy      # waits for healthcheck to pass
    deploy:
      resources:
        limits: { memory: 2G, cpus: "2.0" }
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s

Critical: depends_on vs service_healthy

depends_on: [redis]

Waits for container to start. Redis process may still be initializing. Your API gets ConnectionRefused.

condition: service_healthy

Waits for Redis healthcheck to pass. Requires a HEALTHCHECK defined on the dependency.

Without a healthcheck, Compose hangs indefinitely.

Essential Commands
docker compose up -d          # start all services
docker compose logs -f api    # follow api logs
docker compose ps             # service status
docker compose down           # stop + remove containers
07

Container Security

Default containers are NOT production-safe

What Default Docker Gives You

Default (insecure)
  • Running as root (UID 0)
  • 14 Linux capabilities including NET_RAW, SYS_CHROOT, SETUID
  • Writable filesystem
  • Setuid binaries can escalate privileges
  • Full base image with 100+ CVEs
Production Hardening
  • Non-root user (UID 1000)
  • Drop all capabilities (--cap-drop ALL)
  • Read-only filesystem (--read-only)
  • No-new-privileges flag
  • Trivy CVE scan before push
  • Slim/distroless base image

The Hardened Run Command

docker run -d \
  --cap-drop ALL \                       # remove all 14 default capabilities
  --read-only \                          # immutable filesystem
  --tmpfs /tmp:size=100m \              # writable tmp in RAM
  --tmpfs /app/logs:size=50m \          # writable logs in RAM
  --security-opt no-new-privileges \    # block setuid escalation
  --user 1000:1000 \                    # non-root
  my-api:v1
Scanning with Trivy
trivy image my-api:v1
trivy image --exit-code 1 \
  --severity HIGH,CRITICAL my-api:v1
Secrets — never in ENV

docker inspect and docker history expose all ENV values to anyone with image access. Use BuildKit secret mounts.

Registries & CI/CD

Build → scan → push → sign — automated on every merge

The CI/CD Image Pipeline

Checkout
git clone + secrets via OIDC
Build
docker buildx (multi-arch)
Scan
Trivy — fail on HIGH/CRITICAL
Push
ECR / Artifact Registry
Sign
cosign (keyless OIDC)
Tagging strategy

v1.2.3 + abc123f (git SHA) + branch name

Never deploy :latest to production — it's mutable.

Multi-arch builds

One tag serves linux/amd64 and linux/arm64 via a manifest list. Pull transparently routes to the right architecture.

GitHub Actions — Build, Scan, Push

- uses: docker/setup-buildx-action@v3

- uses: docker/build-push-action@v5
  with:
    platforms: linux/amd64,linux/arm64
    tags: ${{ env.REGISTRY }}/${{ env.IMAGE }}:${{ steps.version.outputs.tag }}
    cache-from: type=gha          # GitHub Actions cache (10 GB)
    cache-to: type=gha,mode=max
    push: true

- uses: aquasecurity/trivy-action@master
  with:
    image-ref: "${{ env.REGISTRY }}/${{ env.IMAGE }}:..."
    exit-code: "1"
    severity: CRITICAL,HIGH

Use OIDC (id-token: write) for authentication — short-lived tokens, no long-lived service account keys stored as secrets.

09

Debugging Containers

A systematic workflow beats 2 AM guessing

The Debugging Workflow

1. docker logs my-api — 90% of answers are here
↓
2. docker exec -it my-api /bin/bash — if container is running
↓
3. docker inspect my-api — exit code, restart count, health status, resource config
↓
4. docker stats my-api — memory near limit? CPU pegged?
↓
5. docker run -it --entrypoint /bin/bash my-api:v1 — override entrypoint, explore manually

Exit Codes — Read the Story

0
Clean exit — check if CMD is a server (it should run forever)
1
Application error — check docker logs for traceback
126
Command not executable — chmod +x your entrypoint script
127
Command not found — binary missing from image
137
OOM kill (SIGKILL) — increase --memory or fix leak
143
Graceful SIGTERM shutdown — normal from orchestrator

The Debug Sidecar Pattern

When your production container has no shell (distroless), attach a debug container to the same network:

# Attach netshoot to same network as production container
docker run -it --rm \
  --network container:my-api \
  nicolaka/netshoot

# From inside netshoot — full debug toolkit, no production image changes
curl localhost:8000/health
dig redis
tcpdump -i eth0 port 5432
ss -tulpn

Debug the production container without modifying the production image.

System Design Patterns

Docker in real ML systems — five production patterns

Five ML System Patterns

Pattern 1 — ML Inference Service
Long-running servers. Multiple API replicas behind nginx. Redis prediction cache. Separate frontend/backend networks.
nginx → FastAPI (×2) → TorchServe/Triton · Redis · PostgreSQL
Pattern 2 — Training Pipeline
Batch jobs, not servers. GPU access via nvidia-docker. service_completed_successfully chains stages like a DAG.
data-prep → train → evaluate · shared volume checkpoints/
Pattern 3 — RAG System
Specialized containers per concern. Query embedding, vector search, and LLM generation each isolated.
FastAPI → sentence-transformers → Qdrant → vLLM

Network Segmentation in Compose

networks:
  frontend:    # client-facing — nginx, api
  backend:     # internal only — postgres, redis, model-server

services:
  nginx: { networks: [frontend] }
  api:   { networks: [frontend, backend] }     # bridge between zones
  postgres: { networks: [backend] }            # never exposed to frontend
  redis:    { networks: [backend] }
  model-server: { networks: [backend] }

Defense-in-depth: even if the API is compromised, an attacker can't reach the database directly from the frontend network.

Key Takeaways

Foundation
  • Containers = isolated processes sharing host kernel
  • Three primitives: namespaces, cgroups, OverlayFS
  • Layer caching: rarely-changing deps first, code last
  • Image size matters: python:3.12-slim for ML
Production Checklist
  • Multi-stage build, non-root user, exec form CMD
  • Custom bridge networks for container DNS
  • service_healthy for reliable Compose startup
  • Trivy scan + cosign sign in CI pipeline
Security
  • --cap-drop ALL --read-only --no-new-privileges
  • Secrets: BuildKit mounts, never ENV/ARG
  • Scan images before every push
Debugging
  • logs → exec → inspect → stats → override entrypoint
  • Exit code 137 = OOM, 143 = graceful stop, 127 = missing binary
  • netshoot sidecar for distroless containers
