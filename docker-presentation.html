01

The Problem Docker Solves

Dependency hell — and the three eras of deployment

Start with the story everyone has lived through as an ML engineer. This lesson answers "why containers exist" at the kernel level.

A Story Every ML Engineer Knows

You train a sentiment classifier on your MacBook. 94% accuracy, sub-50ms inference. You hand it to the platform team. An hour later: "It doesn't work."

The Chain of Failures
  • Their server has Python 3.9 — your code uses match (3.10+)
  • Install 3.11 — NumPy can't find libopenblas
  • Fix that — PyTorch complains about CUDA 11.8 vs 12.1
  • Fix CUDA — six other services on the same server break

Every fix creates a new problem. This is dependency hell.

Interview tip: When asked "why Docker?", don't say "it's a lightweight VM." Say it solves dependency isolation at the OS level while sharing the host kernel. The ML angle (CUDA, PyTorch, system libs) resonates with AI-focused interviewers.

Three Eras of Deployment

Bare Metal

VMs virtualize hardware (hypervisor). Containers virtualize the OS (namespaces + cgroups). That's why containers are 100x faster to start. Interview question: "Are containers more or less secure than VMs?" Answer: less isolated by default (shared kernel), but with proper hardening (Lesson 7) the gap narrows significantly.

How Docker Solves It kernel-level

Namespaces — What You Can See
  • PID — container sees only its own processes
  • NET — own IP address, ports, routing table
  • MNT — own filesystem view
  • UTS — own hostname
  • IPC — own shared memory segments
  • USER — UID 0 in container != UID 0 on host
cgroups — What You Can Use
  • Memory — cap at 2GB, OOM-killed if exceeded
  • CPU — limit to 1.5 cores
  • I/O — throttle disk read/write rates
  • PIDs — cap process count (prevent fork bombs)

Namespaces = isolation. cgroups = resource limits.

This is the "Docker is not a VM" explanation. Docker doesn't emulate hardware. It asks the Linux kernel to create isolated views (namespaces) and enforce resource budgets (cgroups). The container shares the host kernel. That's why a container with Alpine Linux can run on an Ubuntu host.

What Docker Actually Gives You

  • Ship the entire runtime — Python, CUDA, system libs — as one artifact
  • Dev laptop = CI server = production — same binary everywhere
  • No "works on my machine" — the machine IS the container
  • Start in milliseconds, not minutes
  • Run 10 services on one laptop without conflicts
  • ML-specific: pin CUDA version, PyTorch build, and Python version per project
The ML-specific benefit is the strongest argument: CUDA + cuDNN + PyTorch + Python version combinations are a combinatorial nightmare. Docker eliminates this entirely. One Dockerfile = one reproducible environment.

Most engineers can't explain what happens between typing "docker run" and their process starting. This lesson covers the full chain: CLI, daemon, containerd, runc.

Interview insight: Kubernetes doesn't use Docker. It talks directly to containerd (or CRI-O). The "dockershim" was removed in K8s 1.24. But your Dockerfiles and images work everywhere because they follow OCI standards. This is a common senior-level interview question.

What Happens During docker run

1. Pull
Download image layers from registry
2. Create
Allocate filesystem, network, config
3. Start
runc creates namespaces + cgroups
4. Execute
Your CMD/ENTRYPOINT runs as PID 1
# See the full chain in action
docker run --rm python:3.12-slim python -c "print('hello')"

# Behind the scenes:
# 1. CLI sends POST /containers/create to dockerd
# 2. dockerd asks containerd to prepare rootfs from image layers
# 3. containerd asks runc to set up namespaces + cgroups
# 4. runc exec's python with PID 1 inside the new namespace
# 5. "hello" printed to stdout, process exits, container removed (--rm)
runc does the actual kernel-level work: clone() with CLONE_NEWPID, CLONE_NEWNS, CLONE_NEWNET flags to create namespaces, then writes to /sys/fs/cgroup to set resource limits. Your process is the first process in that new namespace, so it gets PID 1.

Why PID 1 Matters critical

Shell Form (BAD)

CMD python main.py

Runs as /bin/sh -c "python main.py". Shell is PID 1. Shell doesn't forward SIGTERM to Python. docker stop waits 10s then SIGKILL.

No graceful shutdown. Requests dropped.

Exec Form (GOOD)

CMD ["python", "main.py"]

Python IS PID 1. Receives SIGTERM directly. Can flush caches, close DB connections, finish in-flight requests.

Graceful shutdown. Zero dropped requests.

This is a favorite interview question: "What's the difference between shell form and exec form in Docker?" The answer is signal handling. In production ML services, dropping inference requests during deploys is unacceptable. Always use exec form.
03

Images and Layers

The packaging model — how images are built, stored, and shared

Understanding layers is the foundation for writing efficient Dockerfiles. Every performance optimization in Lesson 4 depends on understanding this lesson.

An Image Is a Stack of Layers

Each Dockerfile instruction creates one read-only layer. Layers are content-addressable (SHA-256 hash). Identical layers are shared across images.

FROM python:3.12-slim        # Layer 1: base OS (~45MB)
RUN apt-get update && ...     # Layer 2: system packages (~15MB)
COPY requirements.txt .       # Layer 3: requirements file (~1KB)
RUN pip install -r ...        # Layer 4: Python packages (~200MB)
COPY . .                      # Layer 5: your code (~5MB)
Content-Addressable Storage

If two images use the same python:3.12-slim base, they share those layers on disk and in the registry. Pull only downloads missing layers. This is why second pulls are fast.

Layers are stored as tarballs identified by SHA-256 digests. The image manifest maps the sequence of layer digests. This is the OCI Image Specification. Interview tip: "How does Docker save disk space?" = content-addressable layer sharing.

The Union Filesystem

Read-Only Layers (Image)
  • Stacked bottom to top
  • Never modified after creation
  • Shared between all containers using this image
  • Stored once on disk
Writable Layer (Container)
  • Thin layer on top
  • Copy-on-write: read from image, write to container layer
  • Deleted when container is removed
  • Use volumes for persistent data

10 containers from the same image = 1 copy of image layers + 10 thin writable layers. Efficient.

Copy-on-write means reading a file uses the image layer (fast, shared). Writing creates a copy in the container layer. This is why writing to the container filesystem is slower than writing to a volume. For ML model weights, always use volumes, not the writable layer.

Layer Cache — The #1 Build Optimization

Docker caches each layer. If the instruction and all parent layers haven't changed, Docker reuses the cache. Order matters.

BAD — Cache Busted Every Time
COPY . .
RUN pip install -r requirements.txt
# ANY code change busts the cache
# pip install runs EVERY build
GOOD — Cache-Friendly
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
# Code changes only rebuild COPY . .
# pip install is cached!

Rule: copy stable things first (dependencies), volatile things last (code).

The cache invalidation rule: if any instruction changes, ALL subsequent layers are rebuilt. That's why COPY requirements.txt BEFORE COPY . . saves minutes on every build. For ML projects with large deps (PyTorch ~2GB), this is the difference between 15-second and 10-minute builds.

Inspecting Images

# See layers and their sizes
docker history my-api:v1

# IMAGE        CREATED BY                                SIZE
# abc123       CMD ["uvicorn" ...]                       0B
# def456       COPY . .                                  5.2MB
# 789012       RUN pip install --no-cache-dir ...        198MB  <-- biggest layer
# aaa111       COPY requirements.txt .                   1.2KB
# bbb222       RUN apt-get update && ...                 15MB
# ccc333       /bin/sh -c #(nop) FROM python:3.12-slim   45MB

# Dive deeper with dive (third-party tool)
dive my-api:v1
# Shows each layer's filesystem changes and wasted space
The "dive" tool is excellent for finding wasted space in images. Common findings: apt caches not cleaned, pip caches left behind, test files copied into the image, .git directory included. All of these are addressed by a proper .dockerignore and multi-stage builds.
04

Writing Dockerfiles

From bad to production-grade — every line has implications

This is the most practical lesson. We'll take a naive Dockerfile and systematically improve it. The production checklist at the end is your interview reference card.

Multi-Stage Builds essential

Build tools (gcc, g++, headers) should never ship in production images. Multi-stage builds separate build-time from runtime.

# Stage 1: Builder — has gcc, g++, pip compile tools
FROM python:3.12-slim AS builder
WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends gcc g++ libffi-dev \
    && rm -rf /var/lib/apt/lists/*
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Stage 2: Runtime — only runtime libs + compiled packages
FROM python:3.12-slim AS runtime
WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends libpq5 curl \
    && rm -rf /var/lib/apt/lists/*
COPY --from=builder /opt/venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
RUN groupadd --gid 1000 appuser && useradd --uid 1000 --gid appuser appuser
COPY --chown=appuser:appuser . .
USER appuser
HEALTHCHECK --interval=30s --timeout=5s CMD ["curl", "-f", "http://localhost:8000/health"]
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

Image drops from ~1.2GB to ~200MB. Build tools stay in builder stage.

The builder pattern is standard for Python ML services. gcc/g++ are needed to compile numpy, pandas, and other C extensions. But they're not needed at runtime. COPY --from=builder copies only the compiled virtual environment. This is a must-know for interviews.

BuildKit — Cache Mounts & Secret Mounts

# syntax=docker/dockerfile:1

# Cache mount: persist pip's download cache across builds
# NOT included in the image — only on the build host
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt
# First build: 2 min. Add one new package: 15 sec.

# Secret mount: use secrets during build without leaking
# Build: docker build --secret id=pip_token,src=./pip-token.txt .
RUN --mount=type=secret,id=pip_token \
    pip install --extra-index-url \
    https://token:$(cat /run/secrets/pip_token)@pypi.internal/simple/ \
    -r requirements.txt
# Secret NOT stored in image layers. docker history won't show it.
.dockerignore — non-negotiable

Exclude .git, __pycache__, .venv, *.pt, *.onnx, models/, data/, .env. Without it, COPY . . sends gigabytes of model weights to the build daemon.

BuildKit cache mounts are a game-changer for CI. Without them, pip downloads all packages from scratch on every build. With them, only new/changed packages are downloaded. Secret mounts prevent the #1 Docker security mistake: leaking credentials into image layers.

Production Dockerfile Checklist interview-ready

  • Specific, pinned base image: python:3.12.7-slim-bookworm
  • Set PYTHONUNBUFFERED=1 and PYTHONDONTWRITEBYTECODE=1
  • COPY requirements.txt before COPY . . for cache efficiency
  • Use --no-cache-dir for pip (saves 50-200MB)
  • Multi-stage build to separate build-time and runtime dependencies
  • Create and switch to a non-root user
  • Exec form for CMD: ["python", "main.py"] not python main.py
  • Add a HEALTHCHECK instruction
  • Comprehensive .dockerignore (exclude .git, models, data, .env)
  • Clean apt caches in the same RUN layer (rm -rf /var/lib/apt/lists/*)
If an interviewer asks you to review a Dockerfile, go through this checklist systematically. Missing any of these items in production is a red flag. The most commonly missed: non-root user, HEALTHCHECK, and .dockerignore.

Running Containers

Lifecycle, networking, volumes, and resource limits

Building images is half the story. Running them involves ports, networks, volumes, memory limits, restart policies. Each maps to a docker run flag.

Container Lifecycle

# Run (create + start), detached, with a name
docker run -d --name my-api my-image:v1

# Graceful stop: SIGTERM, wait 10s, then SIGKILL
docker stop my-api

# Force kill immediately (no graceful shutdown)
docker kill my-api

# Run and auto-remove on exit (great for one-off ML tasks)
docker run --rm my-image:v1 python -c "print('inference done')"

# Check state
docker inspect my-api --format '{{.State.Status}}'
docker run = docker create + docker start. The --rm flag is essential for batch ML jobs (data processing, evaluation scripts) so you don't accumulate stopped containers. For long-running services, use restart policies instead.

Networking always custom bridge

Network TypeDNSIsolationUse Case
Default bridgeNoMinimalAvoid
Custom bridgeYesPer-networkProduction

# Create a custom network
docker network create ml-services

# Redis resolves by NAME on custom networks
docker run -d --name redis --network ml-services redis:7-alpine
docker run -d --name api --network ml-services -p 8080:8000 my-api:v1

# Inside API container: redis:6379 just works (DNS resolution)
docker exec api python -c "import redis; r = redis.Redis(host='redis'); r.ping()"
The default bridge network is the #1 networking mistake. It has no DNS, so containers can only find each other by IP address, which changes on every restart. Custom bridge networks solve this. In interviews, always mention custom networks when discussing Docker networking.

Volumes & Resource Limits

Named Volume

Docker-managed storage. Survives container removal. Use for DB data, model caches.

-v model-weights:/models:ro

Bind Mount

Maps host directory into container. Great for dev hot-reload.

-v ./src:/app/src

tmpfs

In-memory only. Never touches disk. Use for secrets, temp scratch.

--tmpfs /tmp:size=100m

# Resource limits for ML services
docker run -d \
  --memory 2g \                  # OOM-killed if exceeded
  --cpus 2.0 \                   # max 2 CPU cores
  --gpus all \                   # GPU access (NVIDIA Container Toolkit)
  --restart unless-stopped \     # restart on crash, not on manual stop
  my-api:v1
Interview tip: "What happens if a container exceeds its memory limit?" It gets OOM-killed (exit code 137). "What about CPU?" It gets throttled, not killed. GPU passthrough requires the NVIDIA Container Toolkit and --gpus flag. Named volumes are the right choice for ML model weights.

Docker Compose

Multi-container applications in a single YAML file

Real ML systems have 5-20 containers. Managing them with individual docker run commands doesn't scale. Compose is the answer for single-host orchestration.

ML Inference Stack in One File

services:
  api:
    build: { context: ., target: runtime }
    ports: ["8080:8000"]
    environment:
      - REDIS_URL=redis://redis:6379/0
      - DATABASE_URL=postgresql://app:secret@postgres:5432/mldb
    depends_on:
      redis:    { condition: service_healthy }
      postgres: { condition: service_healthy }
    deploy:
      resources: { limits: { memory: 2G, cpus: "2.0" } }
    restart: unless-stopped

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
    volumes: [redis-data:/data]

  postgres:
    image: postgres:16-alpine
    environment: { POSTGRES_DB: mldb, POSTGRES_USER: app, POSTGRES_PASSWORD: secret }
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d mldb"]
      interval: 10s
    volumes: [pg-data:/var/lib/postgresql/data]

volumes:
  redis-data:
  pg-data:
Key concept: depends_on with condition: service_healthy waits for the healthcheck to pass, not just the container to start. Without this, your API tries to connect to PostgreSQL before it's ready and crashes. This is the #1 Compose mistake in interviews.

depends_on Done Right common mistake

BAD — Container Start
# Only waits for container start
# NOT service readiness!
depends_on:
  - postgres

PostgreSQL takes 2-5s to accept connections after the container starts. API crashes.

GOOD — Health Check
# Waits for healthcheck to pass
depends_on:
  postgres:
    condition: service_healthy

Compose polls the healthcheck. API starts only when PostgreSQL is actually ready.

If you use service_healthy but the dependency has no healthcheck defined, Compose waits forever. Always pair depends_on conditions with healthcheck definitions. Interview question: "How do you handle startup ordering in Docker Compose?" This is the answer.

Profiles, Overrides & Watch

Profiles

Optional services. profiles: ["debug"] only runs with --profile debug. Use for pgAdmin, monitoring.

Override Files

compose.override.yml auto-merges for dev. compose.prod.yml uses -f flag for production.

Compose Watch

File watching. sync for code, rebuild for deps, sync+restart for config changes.

# Essential daily commands
docker compose up -d               # Start all services
docker compose up -d --build       # Rebuild + start
docker compose logs -f api         # Follow logs for one service
docker compose down                # Stop + remove containers
docker compose down -v             # Also delete volumes (destroys data!)
docker compose up -d --scale worker=3  # Scale workers
docker compose watch               # Dev mode with file watching
The override pattern is extremely common in production teams: compose.yml has shared config, compose.override.yml has dev settings (bind mounts, debug ports), compose.prod.yml has production settings (pinned images, resource limits). Compose Watch (2.22+) eliminates manual rebuilds during development.

Container Security

Making containers production-safe — because defaults are NOT secure

A default Docker container runs as root, has 14 Linux capabilities, a writable filesystem, and potentially hundreds of CVEs. This lesson fixes all of that.

Defense in Depth mandatory for prod

# The production-hardened docker run command
docker run -d \
  --user 1000:1000 \                   # Non-root user
  --cap-drop ALL \                     # Drop ALL 14 default capabilities
  --read-only \                        # Read-only filesystem
  --tmpfs /tmp:size=100m \             # Writable tmpfs for temp files
  --security-opt no-new-privileges \   # Prevent privilege escalation
  --memory 2g \                        # Memory limit
  --cpus 2.0 \                         # CPU limit
  my-api:v1
Non-Root User

Root in container = potential root on host (via kernel exploit). Use USER appuser in Dockerfile.

Drop Capabilities

Default containers get NET_RAW, SYS_CHROOT, SETUID. Most apps need zero capabilities.

Interview gold: "How do you secure a Docker container?" Non-root + cap-drop ALL + read-only + no-new-privileges is the minimum. Being able to explain each layer shows senior-level security awareness. Mention that containers share the host kernel, so a kernel exploit + root = host compromise.

Image Scanning & Secrets

Scan with Trivy
# Scan for CVEs
trivy image my-api:v1

# Fail CI on HIGH/CRITICAL
trivy image --exit-code 1 \
  --severity HIGH,CRITICAL \
  my-api:v1

# Scan Dockerfile for misconfig
trivy config Dockerfile
Never Put Secrets in ENV
# Anyone can see ENV values:
docker inspect my-api \
  --format '{{json .Config.Env}}'
# "DB_PASSWORD=exposed!"

# Use secret mounts instead:
docker run -v /secure/db-pass:\
/run/secrets/db_password:ro \
  my-api:v1
  • Scan images in CI — fail on HIGH/CRITICAL CVEs
  • Use BuildKit secret mounts for build-time secrets
  • Mount secret files at runtime, never use ENV/ARG for passwords
  • docker inspect shows all ENV variables to anyone with image access
The ENV secret leak is the most common Docker security mistake. docker inspect, docker history, and even the image manifest expose ENV and ARG values. In CI, always use Trivy or Snyk to gate deployments. In production, use Kubernetes Secrets or a vault.

Security Checklist

  • Run as non-root user (USER directive in Dockerfile)
  • Drop all capabilities (--cap-drop ALL), add back only what's needed
  • Read-only filesystem (--read-only) with tmpfs for writable paths
  • Enable no-new-privileges (--security-opt no-new-privileges)
  • Scan images in CI with Trivy (fail on HIGH/CRITICAL)
  • Use minimal base images (slim or distroless) to reduce attack surface
  • Never put secrets in ENV, ARG, or COPY
  • Pin base images by digest for reproducibility
  • Rebuild weekly to pick up base image security patches
This checklist is your interview reference. Default seccomp and AppArmor profiles are applied automatically by Docker and block ~44 dangerous syscalls. Custom profiles are rarely needed unless you're in a highly regulated environment. The default + this checklist covers 95% of use cases.

Image Registries & CI/CD

Store, distribute, scan, sign — the image supply chain

Images need to get from your laptop to production servers. This lesson covers registries, tagging strategies, multi-arch builds, and CI pipeline design.

Never Deploy :latest

The :latest tag is mutable — pushing overwrites it. Two servers pulling at different times get different images. Use immutable tags: semver (v1.2.3) + git SHA (abc123f).

Tagging strategy matters for rollbacks and audits. Use semver for human readability, git SHA for traceability to exact commit, and digest (sha256:...) for maximum reproducibility. Interview tip: "Why not :latest?" = mutable, untraceable, unreproducible.

Multi-Architecture Builds

Dev on Apple Silicon (arm64). CI on x86 runners. Prod on AWS Graviton (arm64). One tag, all architectures.

# Create multi-arch builder
docker buildx create --name multiarch --driver docker-container --use

# Build for both architectures and push
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t myuser/my-api:v1.2.3 \
  --push .

# Docker creates a manifest list:
# myuser/my-api:v1.2.3 -> linux/amd64: sha256:aaa...
#                       -> linux/arm64: sha256:bbb...
# "docker pull" auto-selects the right architecture
Graviton instances are 20-40% cheaper than x86 on AWS. Multi-arch builds let you use them without separate Dockerfiles. In CI, arm64 builds use QEMU emulation (slow) or native arm64 runners (fast). GitHub Actions now offers arm64 runners.

Debugging Containers

When things go wrong — a systematic workflow

It's 2 AM. PagerDuty fires. Your ML inference service returns 500s. You need a systematic debugging workflow, not random guessing. This lesson gives you that workflow.

The Debugging Flowchart

Container problem?
→
docker logs
→
docker exec
→
docker inspect
→
docker stats
# Step 1: ALWAYS start with logs (solves 90% of problems)
docker logs --tail 100 -f my-api

# Step 2: Get inside the running container
docker exec -it my-api /bin/bash

# Step 3: Check metadata (IP, exit code, restart count, health)
docker inspect my-api --format '{{.State.ExitCode}}'
docker inspect my-api --format '{{.RestartCount}}'

# Step 4: Check resource usage (CPU close to limit? Memory near OOM?)
docker stats --no-stream

# Step 5: Container won't start? Override entrypoint to get a shell
docker run -it --entrypoint /bin/bash my-api:v1
90% of debugging is just reading logs. The entrypoint override trick is critical: when a container crashes immediately on startup, you can't exec into it. Overriding the entrypoint to a shell lets you investigate the filesystem, try running commands manually, and find the issue.

Exit Codes know these cold

0
Clean exit. Process finished successfully. Check if CMD is a script that completed instead of a long-running server.
1
App error. Missing module, config not found, DB connection refused. Check docker logs.
126
Not executable. Entrypoint script missing chmod +x.
127
Command not found. Binary in CMD doesn't exist in the image.
137
SIGKILL (128+9). Usually OOM-killed. Increase --memory limit.
143
SIGTERM (128+15). Graceful shutdown via docker stop or orchestrator.
Exit code 137 is the most common production issue for ML services. Large models consume memory during loading, and if the limit is too low, the container gets OOM-killed. 143 is normal graceful shutdown. 127 means you have a typo in CMD or the binary isn't installed. Interview question: "Your container keeps restarting with exit code 137. What do you do?" = increase memory limit or find the leak.

Debug Sidecar: netshoot

Production containers (distroless, minimal) don't have curl, ping, dig, or tcpdump. Use a debug sidecar instead of bloating your production image.

# Attach to the same network namespace as your container
docker run -it --rm --network container:my-api nicolaka/netshoot

# Now you can:
curl localhost:8000/health          # test API from inside its network
dig redis                           # check DNS resolution
tcpdump -i eth0 port 5432           # capture PostgreSQL traffic
ss -tlnp                            # see listening ports

# Docker events — the audit trail
docker events --filter container=my-api --since 1h
# Shows: die, oom, health_status changes, restarts
Structured Logging = Faster Debugging

Log JSON to stdout: {"level":"error","msg":"DB timeout","host":"postgres","latency_ms":5023}. Pipe through jq to filter. Parseable by ELK, Loki, CloudWatch.

nicolaka/netshoot is the Swiss Army knife of container debugging. It has every network tool you could need. The debug sidecar pattern (attach to same network namespace) lets you debug without modifying the production image. In Kubernetes, this becomes "ephemeral debug containers" (kubectl debug).

Network Segmentation

Frontend (nginx+API) and backend (DB+cache) networks. Defense in depth.

services:
  api:
    build: { context: ., target: runtime }
    deploy: { replicas: 2, resources: { limits: { memory: 1G } } }
    networks: [frontend, backend]
  model-server:
    image: pytorch/torchserve:latest
    volumes: [model-weights:/models:ro]
    deploy: { resources: { limits: { memory: 4G } } }
    networks: [backend]
networks:
  frontend:
  backend:
The two-network pattern (frontend/backend) is critical for interviews. It means nginx can't directly access PostgreSQL — all traffic must go through the API. This is defense-in-depth at the network level. The model server gets 4x the memory of the API because loading models is memory-intensive.

Pattern 2: Training Pipeline

Data Prep
Download, clean, split.
Write to shared volume.
Training
GPU access. Read data,
write checkpoints.
Evaluation
Load model, run metrics.
Push to registry.
services:
  data-prep:
    build: { dockerfile: Dockerfile.data-prep }
    volumes: [training-data:/data]
  train:
    build: { dockerfile: Dockerfile.train }
    volumes: [training-data:/data, checkpoints:/checkpoints]
    deploy:
      resources:
        reservations:
          devices: [{ driver: nvidia, count: 1, capabilities: [gpu] }]
    depends_on:
      data-prep: { condition: service_completed_successfully }
  evaluate:
    build: { dockerfile: Dockerfile.eval }
    depends_on:
      train: { condition: service_completed_successfully }

service_completed_successfully — wait for exit code 0 before starting next stage. Compose as a simple DAG executor.

Training pipelines are batch jobs, not servers. The key Compose feature is service_completed_successfully, which chains stages sequentially. Shared volumes pass data between stages. For production training at scale, you'd use Kubeflow Pipelines or Airflow with KubernetesPodOperator.

start_period Is Critical

LLM container needs start_period: 120s because loading a 16GB model takes 1-2 minutes. Without it, the health check fails during loading and Compose restarts in an infinite loop.

Resource Profiles
  • Embedder: 4GB RAM, CPU-only
  • vLLM: GPU required, 16GB+ RAM
  • Qdrant: 2GB RAM, SSD storage volume
  • API Gateway: 512MB, fast and light
RAG systems are the most common AI pattern in 2024-2025. Each container has dramatically different resource requirements. The start_period tip is gold for interviews — it shows you've actually deployed ML models in containers and dealt with the startup time problem.
