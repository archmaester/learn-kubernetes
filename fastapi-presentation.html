Why FastAPI for AI Engineering

The gap between training a model and serving it over HTTP

This lesson explains why FastAPI won the ML serving framework race. The key argument: model serving is infrastructure, not an afterthought. A Flask endpoint bolted on in a Friday afternoon will fall over at 50 concurrent requests.

The ML Serving Gap

You train a document classifier. 94% accuracy, sub-80ms inference. You hand it to the platform team. They bolt on a Flask endpoint. Result:

What Goes Wrong
  • Falls over at 50 concurrent requests — Flask is sync, one thread per request
  • Returns cryptic 500 errors on malformed input — no validation
  • No auto-generated documentation — clients guess the contract
  • Leaks memory because nobody thought about GIL + model inference

Model serving is infrastructure. It deserves the same engineering rigor as the model itself.

Interview tip: When asked "why FastAPI over Flask for ML?", the answer is: native async (don't block on I/O), automatic OpenAPI docs (clients self-serve), Pydantic validation (garbage in, structured error out), and dependency injection (clean resource management for DB/model/cache). It's not about speed benchmarks — it's about engineering correctness.

Async Python Deep Dive

The event loop, the GIL, and why async matters for ML APIs

Most Python developers use async without understanding it. This lesson covers the event loop from first principles: what it actually does, why the GIL doesn't protect you, and when async helps vs when it hurts. Critical for interviews.

The Event Loop Model

1. Call async fn
Returns a coroutine
(not a result)
2. await
Suspends coroutine,
yields control to loop
3. I/O completes
OS notifies loop
via epoll/kqueue
4. Resume
Coroutine picks up
where it left off
# One thread, many concurrent requests
async def classify(text: str) -> dict:
    # CPU work: tokenize (runs on the single thread)
    tokens = tokenize(text)

    # I/O: check Redis cache (suspends, loop handles other requests)
    cached = await redis.get(f"classify:{hash(text)}")
    if cached:
        return json.loads(cached)

    # CPU work: model inference (blocks the loop!)
    result = model.predict(tokens)  # BAD if slow

    # I/O: cache result (suspends again)
    await redis.setex(f"classify:{hash(text)}", 3600, json.dumps(result))
    return result
The event loop is a single-threaded scheduler. During await, the coroutine suspends and the loop runs other coroutines. This is how one thread handles thousands of concurrent connections. But CPU-bound work (model inference) blocks the loop. This is the #1 async mistake in ML services.

The GIL Doesn't Save You common trap

Blocking the Event Loop
# 500ms model inference blocks
# ALL other requests wait
result = model.predict(tokens)
# 1000 concurrent users?
# They all queue behind this.
Offload to Thread Pool
# Run CPU work in a thread
import asyncio
result = await asyncio.to_thread(
    model.predict, tokens
)
# Loop stays free for other
# requests while this runs

Rule: await for I/O, asyncio.to_thread() for CPU, ProcessPoolExecutor for heavy compute.

Interview question: "Your async FastAPI endpoint calls a PyTorch model. Under load, latency spikes for all endpoints. Why?" Answer: model inference is CPU-bound and blocks the event loop. Solution: asyncio.to_thread() or ProcessPoolExecutor. The GIL means threads don't give true parallelism for pure Python, but C extensions like NumPy/PyTorch release the GIL during compute.

uvloop is a drop-in replacement for asyncio's event loop written in Cython on top of libuv. It reduces event loop overhead by 2-4x. In production FastAPI deployments, always install uvloop and httptools. Uvicorn auto-detects both.

Pydantic V2 — Data Validation at Scale

Type-safe contracts between your API and every client

Pydantic V2 is a complete rewrite with a Rust core (pydantic-core). It's 5-50x faster than V1 and is the backbone of FastAPI's validation, serialization, and OpenAPI generation. Schemas are the contract — change them carefully.

IntelliAPI: Document Schema

from pydantic import BaseModel, ConfigDict, Field, field_validator, computed_field
from enum import Enum
from datetime import datetime

class DocumentCategory(str, Enum):
    LEGAL_CONTRACT = "legal_contract"
    FINANCIAL_REPORT = "financial_report"
    MEDICAL_RECORD = "medical_record"
    TECHNICAL_SPEC = "technical_spec"

class DocumentRequest(BaseModel):
    model_config = ConfigDict(str_strip_whitespace=True)

    text: str = Field(..., min_length=10, max_length=100_000,
                      description="Document text to classify")
    language: str = Field(default="en", pattern=r"^[a-z]{2}$")
    priority: str = Field(default="normal")

    @field_validator("text")
    @classmethod
    def text_not_empty_after_strip(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("text must contain non-whitespace characters")
        return v
This schema does six things: strips whitespace, validates min/max length, enforces a regex pattern on language, provides defaults, generates OpenAPI docs, and runs custom validators. All from type annotations. Interview tip: "How do you validate API input in FastAPI?" Answer: Pydantic models. They validate, serialize, document, and generate OpenAPI specs from a single class definition.

Response Models & Computed Fields

class ClassifyResponse(BaseModel):
    model_config = ConfigDict(from_attributes=True)  # ORM mode

    document_id: str
    label: DocumentCategory
    confidence: float = Field(..., ge=0.0, le=1.0)
    model_version: str
    processing_time_ms: float
    tokens_used: int

    @computed_field
    @property
    def is_high_confidence(self) -> bool:
        return self.confidence >= 0.85
from_attributes=True

Allows creating the model from ORM objects: ClassifyResponse.model_validate(db_row). Essential for SQLAlchemy integration.

computed_field

Derived fields that appear in serialization and OpenAPI but aren't stored. Perfect for is_high_confidence, word_count, etc.

Response models control what leaves your API. They prevent leaking internal fields (like raw DB IDs or internal model metadata). computed_field is new in Pydantic V2 — it replaces the V1 pattern of using a validator to set a field. Interview question: "How do you prevent leaking internal data?" Answer: separate request and response Pydantic models.

Dependency Injection & Application Lifecycle

Clean resource management for databases, models, and caches

Dependency injection is FastAPI's most powerful feature and most misunderstood. It's not just about injecting config — it manages resource lifecycles (DB sessions, Redis connections, ML models) with automatic cleanup.

Depends() in Action

from fastapi import APIRouter, Depends
from sqlalchemy.ext.asyncio import AsyncSession
from onnxruntime import InferenceSession
import redis.asyncio as redis

router = APIRouter(prefix="/classify", tags=["classification"])

@router.post("/", response_model=ClassifyResponse)
async def classify_document(
    request: ClassifyRequest,
    user: User = Depends(get_current_user),    # Auth
    db: AsyncSession = Depends(get_db),         # DB session
    model: InferenceSession = Depends(get_classifier),  # ML model
    cache: redis.Redis = Depends(get_redis),    # Cache client
) -> ClassifyResponse:
    """Four dependencies injected — all managed by FastAPI."""
    cache_key = f"classify:{hashlib.sha256(request.text.encode()).hexdigest()}"
    cached = await cache.get(cache_key)
    if cached:
        return ClassifyResponse(**json.loads(cached))
    result = await asyncio.to_thread(model.run, None, {"input": tokens})
    await cache.setex(cache_key, 3600, json.dumps(result))
    return ClassifyResponse(**result)
This route has four dependencies: auth, database, ML model, and cache. Each is injected automatically, each has its own lifecycle. The route only contains business logic. Interview tip: "How do you manage database sessions in FastAPI?" Answer: yield dependencies with Depends(). FastAPI manages the session lifecycle — open before the route, close/rollback after.

Yield Dependencies essential

from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker

async def get_db() -> AsyncGenerator[AsyncSession, None]:
    """Yield a DB session — auto-closed after the request."""
    async with async_session_factory() as session:
        try:
            yield session        # Route handler runs here
            await session.commit()
        except Exception:
            await session.rollback()
            raise
        finally:
            await session.close()
Per-Request Caching

If get_db appears in both get_current_user and the route, FastAPI resolves it once per request and reuses the same session. No duplicate connections.

Yield dependencies are context managers in disguise. Code before yield = setup. Code after yield = teardown. FastAPI guarantees teardown runs even if the route raises an exception. This is how you prevent connection leaks in production. Interview question: "How do you ensure DB connections are returned to the pool?" Answer: yield dependencies with try/finally.

Lifespan: App-Level Resources

from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    # STARTUP: load resources once, shared across all requests
    app.state.model = onnxruntime.InferenceSession("model.onnx")
    app.state.redis = redis.from_url(settings.redis_url)
    engine = create_async_engine(settings.database_url)

    yield  # App runs here

    # SHUTDOWN: clean up
    await app.state.redis.close()
    await engine.dispose()

app = FastAPI(title="IntelliAPI", lifespan=lifespan)
Replaces deprecated @app.on_event

The lifespan context manager replaces @app.on_event("startup") and @app.on_event("shutdown"). It's cleaner and allows sharing state between startup and shutdown.

Lifespan events handle app-level resources: ML models, DB engines, Redis pools. These are loaded once at startup and shared across all requests via app.state. Per-request resources (DB sessions) use yield dependencies. Interview tip: clearly distinguish app-level (lifespan) from request-level (Depends) resource management.

Authentication & Authorization

JWT, OAuth2, RBAC — securing IntelliAPI endpoints

Every production API needs auth. FastAPI provides OAuth2 security schemes that integrate with OpenAPI (the Swagger UI gets a lock icon and a login form). This lesson covers JWT-based auth with role-based access control.

from fastapi import Depends, HTTPException, status
from fastapi.security import OAuth2PasswordBearer
from jose import jwt, JWTError

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/auth/token")

async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_db),
) -> User:
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=["HS256"])
        email: str = payload.get("sub")
        if email is None:
            raise HTTPException(status_code=401, detail="Invalid token")
    except JWTError:
        raise HTTPException(status_code=401, detail="Token expired or invalid")
    user = await db.execute(select(User).where(User.email == email))
    return user.scalar_one_or_none() or raise_401()
OAuth2PasswordBearer tells FastAPI to look for a Bearer token in the Authorization header. It also adds a lock icon to Swagger UI. The get_current_user dependency is reusable — add Depends(get_current_user) to any route to protect it. Interview tip: explain the difference between authentication (who are you?) and authorization (what can you do?).

Role-Based Access Control RBAC

from enum import Enum
from functools import wraps

class Role(str, Enum):
    VIEWER = "viewer"     # Read-only access
    ANALYST = "analyst"   # Classify documents
    ADMIN = "admin"       # Full access + model management

def require_role(*allowed_roles: Role):
    """Dependency factory — returns a dependency that checks the role."""
    async def role_checker(user: User = Depends(get_current_user)):
        if user.role not in allowed_roles:
            raise HTTPException(
                status_code=403,
                detail=f"Role '{user.role}' not in {allowed_roles}"
            )
        return user
    return role_checker

# Usage in routes:
@router.post("/classify")
async def classify(user=Depends(require_role(Role.ANALYST, Role.ADMIN))):
    ...

@router.delete("/admin/purge-cache")
async def purge(user=Depends(require_role(Role.ADMIN))):
    ...
The require_role factory creates a parameterized dependency. This pattern is cleaner than decorators because it integrates with FastAPI's DI system and appears in OpenAPI docs. Interview question: "How would you implement RBAC in FastAPI?" This is the answer: a dependency factory that returns a checker function.

Auth Security Checklist

  • Hash passwords with bcrypt (never SHA-256 — bcrypt is intentionally slow)
  • Short-lived access tokens (15-30 min) + long-lived refresh tokens
  • Store JWT secret in environment variable, never in code
  • Validate exp, iss, aud claims on every request
  • Rate-limit login endpoints (prevent brute force)
  • Return 401 for auth failures, 403 for authorization failures
  • Never put passwords or tokens in URL query parameters
  • Never store JWTs in localStorage (XSS vulnerable) — use httpOnly cookies
The 401 vs 403 distinction trips up many developers. 401 = "I don't know who you are" (missing/invalid token). 403 = "I know who you are but you don't have permission." Getting this right in your API shows attention to HTTP semantics. The localStorage warning is a real security issue — XSS can steal tokens. httpOnly cookies are the safe default for browser clients.

Middleware, Error Handling & Structured Logging

Cross-cutting concerns that make IntelliAPI production-ready

Middleware, error handlers, and logging are the "invisible infrastructure" of a production API. They run on every request, shape every error response, and generate every log line. Getting them right is the difference between debugging in 5 minutes vs 5 hours.

def create_app() -> FastAPI:
    app = FastAPI(title="IntelliAPI", lifespan=lifespan)

    # Middleware order: LAST added = OUTERMOST = runs FIRST
    app.add_middleware(RateLimitMiddleware, rpm=100)     # 4. innermost
    app.add_middleware(LoggingContextMiddleware)          # 3.
    app.add_middleware(TimingMiddleware)                  # 2.
    app.add_middleware(RequestIDMiddleware)               # 1.
    app.add_middleware(CORSMiddleware,                    # 0. outermost
        allow_origins=["https://app.intelliapi.com"],
        allow_credentials=True,
        allow_methods=["*"], allow_headers=["*"],
        expose_headers=["X-Request-ID", "X-Process-Time"])
    return app

CORS must be outermost — it must handle OPTIONS preflight before auth rejects it.

The middleware order is a common interview question. Last added = outermost = runs first. CORS handles preflight OPTIONS requests that don't have auth headers, so it must run before any auth middleware. Request ID must run before logging so the ID is available in log context. Getting the order wrong causes subtle bugs like missing CORS headers on error responses.

Structured Logging with structlog

import structlog

# Middleware binds context to all logs in the request
structlog.contextvars.bind_contextvars(
    request_id=request.state.request_id,
    method=request.method,
    path=request.url.path,
)

# Any code in the request chain gets context automatically
logger = structlog.get_logger()
logger.info("classification_started", model="v2", text_length=1523)

# Output (JSON in production):
# {"event": "classification_started", "model": "v2", "text_length": 1523,
#  "request_id": "abc-123", "method": "POST", "path": "/classify",
#  "level": "info", "timestamp": "2024-01-15T10:30:00Z"}
Why Structured Logging?

JSON logs are queryable: request_id:abc-123 AND level:error. Text logs require regex. At 10K RPM, the difference between 5-second and 5-minute debugging.

structlog's contextvars integration is the key. Bind request_id once in middleware, and every log call anywhere in the request chain includes it automatically. No passing request_id through function parameters. In production, pipe JSON logs to ELK, Datadog, or CloudWatch. Interview tip: "How do you debug issues in production?" Answer: structured logging with request_id correlation.

Exception Handlers

from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse

class IntelliAPIError(Exception):
    def __init__(self, code: str, detail: str, status: int = 400):
        self.code = code; self.detail = detail; self.status = status

@app.exception_handler(IntelliAPIError)
async def handle_business_error(request: Request, exc: IntelliAPIError):
    return JSONResponse(status_code=exc.status, content={
        "error": exc.code,
        "detail": exc.detail,
        "request_id": getattr(request.state, "request_id", None),
    })

@app.exception_handler(Exception)
async def handle_unhandled(request: Request, exc: Exception):
    logger.error("unhandled_error", exc_info=exc)  # Full traceback logged
    return JSONResponse(status_code=500, content={
        "error": "internal_error",
        "detail": "An unexpected error occurred",  # Safe message to client
        "request_id": getattr(request.state, "request_id", None),
    })
Two key principles: (1) Business errors (IntelliAPIError) return structured error codes that clients can programmatically handle. (2) Unhandled errors log the full traceback but return a safe generic message. Never leak stack traces to clients. Always include request_id in error responses so clients can report it for debugging.

Testing FastAPI Applications

The testing pyramid, dependency overrides, and testcontainers

FastAPI's architecture — DI, Pydantic, async-first — makes it one of the most testable Python frameworks. The secret weapon is dependency_overrides: swap any dependency (DB, ML model, external API) with a fake for testing.

The Testing Pyramid

Unit Tests (Many)
  • Pure functions, validators
  • Pydantic model validation
  • Dependency overrides for fakes
  • Fast: <1ms per test
Integration (Some)
  • Real DB via testcontainers
  • Full request-response cycle
  • Auth + validation + persistence
  • Slower: 50-200ms per test
E2E (Few)
  • Live server + real deps
  • Contract verification
  • Slowest, most brittle
  • Only for critical paths

Golden rule: test behavior, not implementation. Your tests should survive a refactor.

Interview question: "What's your testing strategy for a FastAPI microservice?" Answer: testing pyramid. Many fast unit tests with dependency overrides and fake services. Some integration tests with testcontainers (real Postgres). Few E2E tests. Target 80%+ coverage on business logic. Always test auth, validation, and error handling.

Dependency Overrides key pattern

# Fake classifier — deterministic, no ML model needed
class FakeClassifier:
    async def classify(self, text: str) -> dict:
        return {"label": "contract", "confidence": 0.92, "model_version": "test-v1"}

# Swap real classifier with fake in tests
@pytest.fixture(autouse=True)
def use_fake_classifier():
    app.dependency_overrides[ClassifierService] = FakeClassifier
    yield
    app.dependency_overrides.pop(ClassifierService, None)

# Test the route — classifier is faked, but everything else is real
@pytest.mark.asyncio
async def test_classify_returns_label(auth_client: AsyncClient):
    response = await auth_client.post(
        "/api/v1/classify",
        json={"text": "Service Agreement between Acme Corp..."},
    )
    assert response.status_code == 200
    assert response.json()["label"] == "contract"
    assert 0 <= response.json()["confidence"] <= 1
dependency_overrides is a dict mapping original dependency to replacement. The replacement must have a compatible interface. Always clean up overrides after tests to prevent state leakage. The FakeClassifier pattern is how you test API behavior without loading a 500MB ML model in every test run.

conftest.py: Test Foundation

# tests/conftest.py — shared fixtures for the entire test suite
import pytest_asyncio
from httpx import ASGITransport, AsyncClient
from app.main import app
from app.auth import create_access_token

@pytest_asyncio.fixture
async def client(db_session) -> AsyncGenerator[AsyncClient, None]:
    async def _override_get_db():
        yield db_session
    app.dependency_overrides[get_db] = _override_get_db
    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://testserver") as ac:
        yield ac
    app.dependency_overrides.clear()

@pytest_asyncio.fixture
async def auth_client(client) -> AsyncClient:

httpx.AsyncClient with ASGITransport talks directly to FastAPI — no server needed.

The key insight: AsyncClient with ASGITransport speaks ASGI directly to your app. No HTTP server is started. This is fast and reliable. The auth_client fixture adds a JWT so all protected endpoints work without per-test auth setup. Interview question: "TestClient vs AsyncClient?" TestClient is sync (Starlette), AsyncClient is async-native (httpx) and required for async fixtures.

Background Tasks, WebSockets & Streaming

Async patterns beyond request-response

Not every operation fits in a request-response cycle. This lesson covers three patterns: BackgroundTasks (fire-and-forget), SSE (one-way streaming), and WebSockets (bidirectional). Each has specific use cases in ML serving.

BackgroundTasks: Fire and Forget

@router.post("/classify", response_model=ClassifyResponse)
async def classify_document(
    body: ClassifyRequest,
    background_tasks: BackgroundTasks,
    classifier: ClassifierService = Depends(),
    audit: AuditService = Depends(),
    user=Depends(get_current_user),
):
    # Client waits for this
    result = await classifier.classify(body.text)

    # Runs AFTER response is sent — client doesn't wait
    background_tasks.add_task(write_audit_log, audit, user.email, result)
    background_tasks.add_task(send_webhook, "https://hooks.example.com", result)

    return ClassifyResponse(**result)
PatternUse WhenLimitations
BackgroundTasksFire-and-forget, <30s, non-criticalNo retry, dies with process
Celery + RedisRetries needed, long-running, must survive restartsSeparate worker, complex setup
arqAsync-native Celery alternativeSmaller ecosystem
BackgroundTasks run in-process after the response is sent. They're perfect for audit logs, webhooks, cache warming. But they have no retry, no persistence, and die if the worker crashes. For anything critical (payment processing, email delivery), use Celery. Interview question: "When would you use BackgroundTasks vs Celery?" This table is the answer.

SSE: Streaming LLM Output ML essential

from sse_starlette.sse import EventSourceResponse

async def classify_stream(text: str, classifier) -> AsyncGenerator[str, None]:
    yield json.dumps({"event": "progress", "stage": "tokenizing", "percent": 10})
    await asyncio.sleep(0.3)

    yield json.dumps({"event": "progress", "stage": "inference", "percent": 40})
    result = await classifier.classify(text)

    yield json.dumps({"event": "complete", "percent": 100, "result": result})

@router.post("/classify/stream")
async def classify_with_progress(request: Request, body: dict, ...):
    async def event_generator():
        async for chunk in classify_stream(body["text"], classifier):
            if await request.is_disconnected():
                break
            yield chunk
    return EventSourceResponse(event_generator())
SSE vs WebSocket

SSE = one-way (server to client). Auto-reconnects, works through proxies, standard HTTP. Use for LLM token streaming, progress updates. WebSocket = bidirectional. Use for chat, collaborative editing.

SSE is the standard pattern for streaming LLM output. OpenAI's API uses SSE. The key detail: always check request.is_disconnected() in the generator loop to stop work when the client disconnects. Without this check, you waste GPU compute generating tokens nobody will receive.

WebSocket: Interactive Classification

class ConnectionManager:
    def __init__(self):
        self.active: dict[str, WebSocket] = {}

    async def connect(self, user_id: str, ws: WebSocket):
        await ws.accept()
        self.active[user_id] = ws

    async def broadcast(self, data: dict):
        for ws in self.active.values():
            await ws.send_json(data)

@router.websocket("/ws/classify")
async def websocket_classify(ws: WebSocket, classifier=Depends()):
    user_id = await ws_authenticate(ws)
    if not user_id:
        await ws.close(code=4001, reason="Auth failed")
        return
    await manager.connect(user_id, ws)
    try:
        while True:
            data = await ws.receive_json()
            if data.get("type") == "classify":
                result = await classifier.classify(data["text"])
                await ws.send_json({"type": "result", **result})
    except WebSocketDisconnect:
        manager.disconnect(user_id)
WebSocket auth is different from HTTP auth. You can't use Authorization headers. Three patterns: (1) token in query params, (2) token in first message, (3) session cookies. Behind a load balancer, WebSocket connections are sticky to one instance. Use Redis pub/sub to broadcast across instances. Interview tip: mention the sticky connection problem — it shows you've thought about scale.

Gunicorn is the process manager; each Uvicorn worker runs its own async event loop. A crash in one worker doesn't affect others, and Gunicorn auto-restarts failed workers. Key gotcha for ML: if each worker loads a 500MB model, 4 workers = 2GB RAM. For GPU inference, use 1 worker per GPU and rely on the async event loop for concurrency.

If liveness calls the DB and the DB is slow, Kubernetes kills healthy containers. Keep liveness trivial.

The liveness vs readiness distinction is a top interview question. Liveness must be trivial (return 200, no deps). If it accidentally depends on the database and the DB is slow, Kubernetes kills the container, which increases DB load, which makes other containers fail their liveness checks — cascading failure. Readiness checks every dependency. Startup probes are essential for ML model loading (60+ seconds).

Graceful shutdown order matters: stop accepting new work, drain in-flight requests, close connections. If you close connections first, pending queries fail. The Prometheus metrics middleware must normalize URL paths — /documents/123 becomes /documents/{id} — to prevent cardinality explosion. Every unique label combination is a time series. Two-path architecture is the key insight. Classification is fast enough for synchronous response. Summarization is too slow — holding HTTP connections for 5+ seconds wastes API server resources. The async path returns 202 immediately and the client polls for results. This is the standard pattern for mixed-latency AI systems.

Dynamic Batching 10-15x throughput

class DynamicBatcher:
    """Collect individual requests into GPU batches."""

    def __init__(self, model_fn, max_batch_size=32, max_wait_ms=10.0):
        self.model_fn = model_fn
        self.max_batch_size = max_batch_size
        self.max_wait_ms = max_wait_ms
        self._queue: asyncio.Queue[BatchItem] = asyncio.Queue()

    async def submit(self, input_data, timeout=30.0) -> Any:
        """Submit one request — returns when the batch result is ready."""
        item = BatchItem(input_data=input_data)
        await self._queue.put(item)
        return await asyncio.wait_for(item.future, timeout=timeout)

    async def _collect_batch(self) -> list[BatchItem]:
        """Wait for items until batch is full OR max_wait_ms expires."""
        batch = [await self._queue.get()]  # Wait for at least one
        deadline = time.time() + (self.max_wait_ms / 1000)
        while len(batch) < self.max_batch_size and time.time() < deadline:
            try:
                item = await asyncio.wait_for(self._queue.get(), deadline - time.time())
                batch.append(item)
            except asyncio.TimeoutError:
                break
        return batch

A batch of 32 items is only 2-3x slower than a single item on a GPU. Batching = 10-15x throughput gain.

This is the single most impactful optimization for GPU inference. Each caller gets an asyncio.Future for their individual result. The batcher waits up to 10ms to collect items, then runs one forward pass. The 10ms collection latency is invisible to users but allows much larger batches. Interview gold: explain why batching works (GPU parallelism) and the latency-throughput tradeoff (max_wait_ms).

Resilience Patterns

Circuit Breaker
  • CLOSED → OPEN after 5 failures
  • OPEN: fail fast with 503
  • HALF-OPEN: test with 1 request
  • Per-service: circuit_db, circuit_model
Load Shedding
  • 80% capacity: drop free-tier
  • 90% capacity: drop all except critical
  • Health checks always pass through
  • Return 503, not 429
Result Caching
  • Content-addressable: hash(text + model_version)
  • Same document = same classification
  • 30-40% cache hit rate in enterprise
  • Each hit saves one GPU inference
Three resilience patterns that work together. Circuit breaker prevents cascading failures when a downstream service is dying. Load shedding protects the system under extreme load by dropping low-priority requests first. Result caching eliminates redundant GPU compute. Together they make the system robust under failure and overload. Interview tip: describe all three as layers of defense.
